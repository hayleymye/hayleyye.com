Field Mapping

Merge, 2024

Empowering enterprise users with flexible, self-serve data mapping across integrations.

Project Overview & Introduction

In 2022, Merge faced a key integration challenge: how to handle non-standard data across diverse third-party platforms.

Accessing custom data is critical for customers with enterprise end users because they rely on systems with specialized data needs that go beyond Merge‚Äôs Common Models. Enterprise platforms are rarely one-size-fits-all, and having the flexibility to extend or modify data fields allows businesses to tailor integrations to their specific workflows.

To address the limited flexibility of Merge‚Äôs standardized data models, we launched a beta version of Field Mapping in 2023 to a few selected customers. These customers gave us important feedback and insight into how we could make Field Mapping more user-friendly and comprehensive. Through iteration and redesign, this feature grew into a full-fledged Field Mapping platform that is a flagship feature today and is used by over half of Merge‚Äôs Professional and Enterprise organizations and supports 6,000+ mapping instances across (as of November 2024).

Although I led the design of Field Mapping from its conception in 2022 , this case study will focus on the latest iterations of the feature from 2024.

The Challenge

Merge‚Äôs Common Models (like Employee, Ticket, Candidate, etc.) standardize data across platforms such as BambooHR, Zendesk, and Greenhouse. But every platform‚Äîand often every company‚Äîhas custom fields unique to their use case, such as:

  • ‚ÄúT-shirt size‚Äù on an employee record
  • ‚ÄúUrgency score‚Äù on a support ticket
  • ‚ÄúLead source‚Äù on a CRM contact

Without support for custom fields:

  • This data is inaccessible through Merge.
  • Integrators must build custom logic outside of Merge.
  • It breaks the promise of a ‚Äúunified API‚Äù for real-world use cases.

Our goal was to make custom data mapping intuitive, accurate, and scalable‚Äîwhether users were mapping an employee‚Äôs personal email or transforming nested JSON fields across platforms like Workday, BambooHR, and Salesforce.

We collaborated with Merge‚Äôs Field Mapping power users to conduct workflow observations and analyze support tickets to identify pain points.

Key Problems We Identified:

  1. There were too many clicks required to create a mapping.
  1. Customers had built thousands of mappings manually, a sign of high need but also high friction. We found that many use cases require custom fields that are the same across an integration.
  1. End users couldn‚Äôt easily self-configure mappings because the Merge Link UI (end-user interface) for Field Mapping was outdated and unintuitive, blocking them from customizing integrations and limiting the feature‚Äôs scalability.
  1. Users did not feel confident in choosing the right third-party field. Not all integrations support human-readable field names (for example, a typical field name in Workday: wd:Worker_Data.wd:Employment_Data.wd:Worker_Job_Data.wd:Position_Data.wd:Job_Profile_Summary_Data.wd:Management_Level_Reference) and companies often deprecate and replace fields as they grow. Field names can be unclear, and the field type and endpoint were not enough context for users to trust their selections. Users would map a field, wait for data re-sync to see what value was returned, and then update the mapping if the value was not what they expected.
  1. Customers struggled to map nested or complex fields (e.g., List[object]), requiring them to build custom logic outside of Merge, which is the problem Field Mapping (and Merge‚Äôs Unified APIs overall) sought to address in the very beginning. This issue was particularly common in enterprise applications like Workday, NetSuite, and Salesforce, where custom objects can be complex and extensive, making it difficult to isolate specific fields.

Ideation & Solution

Design prototypes were tested with feedback loops across engineering, post-sales, and selected customers. After some iteration, we came up with the following solutions to address the key problems identified above.

1. Simplified mapping flow

  • We reimagined the mapping flow to reduce friction and better fit the user journey.
  • Users start by choosing the type of mapping they want to create ‚Üí then selecting the remote field they want ‚Üí then selecting or creating a target Common Model field ‚Üí then validating their selection.

2. Integration-wide mappings

  • We introduced the ability to create integration-wide mappings (vs. Linked Account-specific) to improve scalability.
  • Users can create a mapping that will be automatically applied to all new Linked Accounts for a given integration.

üñºÔ∏è¬†[screenshot of integration wide mapping]

3. Wide Merge Link interface

  • After successfully implementing a wider screen for another Merge Link feature, we applied the same approach to Field Mapping to display more information, making the experience more self-serve.
  • We overhauled the Link interface to allow for more intuitive left-to-right mapping (like in the dashboard) instead of vertical mapping.

üñºÔ∏è¬†[screenshot of wide link UI]

4. Remote Field coverage and example values

  • We added more information to the Remote Fields dropdown to increase confidence and provide users with more context about each field before they create and commit to a mapping.
  • Coverage is a percentage value that provides visibility into how often a Remote Field appears across synced objects. The higher the percentage, the more likely the field is what the user is looking for.
  • Example values provide visibility into what data is actually going to be synced. If the value is null or unexpected, the field is likely not what the user is looking for.

üñºÔ∏è¬†[screenshot of remote field dropdown]

5. Preview values

  • We added a small input to allow users to instantly preview real values for each mapping in the dashboard to easily double check their mappings at a glance. The values would replace the endpoint so that the visual change would not cause each card‚Äôs height to increase jarringly.
  • I originally wanted the input to be a dropdown of human-readable objects that users could scroll through and select, to eliminate the need to retrieve an object ID and reduce errors, but we were limited by engineering resources and this solution addressed the customer need while also being easy to implement.

üñºÔ∏è¬†[gif of preview values prototype]

6. Advanced Mapping with JMESPath

  • We added an optional Advanced Mapping step to the mapping flow to enable users to access nested fields and apply unique business logic within the dashboard
  • We decided to use JMESPath, a query language for JSON, because even though most customer use cases could be solved by indexing on the List[object] , the order can change so mapping via index can cause errors.
  • We implemented an easy click-to-generate query functionality to simplify the Advanced Mapping experience for most users, but still gave users the flexibility to write their own complex queries if necessary.

Insights & Impact

What We Learned:

  • Providing clear, contextual information builds user trust and reduces mapping errors:
    • Feature enhancements like coverage indicators and example values increased user confidence when selecting fields.
    • Being able to see real data from mappings further increased user confidence.
  • Simplify the experience for regular users, and allow advanced configurations for power users:
    • The streamlined mapping flow allows users to more easily and quickly create mappings.
    • For those willing to learn, using a query language like JMESPath makes Field Mapping flexible and powerful enough for complex use cases.
  • Change is good‚Äîthings don‚Äôt have to remain the way they have always been:
    • The initial design of Field Mapping in Merge Link was narrow simply because that was what we were used to, not because it was the best user experience.
    • After testing the wider Link UI and talking to customers, it proved to be a much better experience and no one even mentioned anything about the wider screen being unfamiliar.

Results:

  • Field Mapping usage grew by 50% from Q1 to Q3 2024.
  • Reduced support tickets related to incorrect mappings and syncing errors.
  • One customer saw a 2x increase in Field Mapping adoption after UI improvements.
  • The enhanced Field Mapping feature made Merge more appealing to enterprise clients seeking customizable integration solutions.

Final Thoughts

Field Mapping is perhaps the most meaningful project of my career so far. I started brainstorming for the initial concept just a few months after joining Merge in 2022 and continued iterating on it through each update, until the most recent one in December 2024. As I grew as a designer, I constantly found new ways to improve the feature, iterating on both the interface and the user experience.

Reflecting on the process, I would have dedicated more time to user testing during the beta phase. Due to tight deadlines, we prioritized shipping quickly, relying on hypotheses rather than user validation. Although the final version is successful, gathering feedback earlier could have accelerated these improvements.

This project not only greatly enhanced Merge‚Äôs product offering but also deepened my ability to design intuitive, scalable solutions for complex data scenarios. Leading Field Mapping from concept to flagship feature taught me how to build products that empower enterprise users.
